9.1 分類問題の性能指標¶

本章のテーマは､性能評価とチューニングです｡前回のSVMで､ハイパーパラメータを何かしらの｢ちょうどいい値｣に定めることが必要になりました｡今回は､その｢ちょうどいい値｣を自動的に見つけるためのグリッドサーチという手法について学びます｡ハイパーパラメータの値の様々な組み合わせに対して学習を行い､最適な組み合わせを探索的に見つけます｡また､そのための前提知識となる分類問題の性能指標や､より信頼性の高い汎化性能の評価方法を学びます｡

汎化性能を評価するにあたって､まずはその評価指標を正しく理解する必要があります｡特に､分類問題には回帰問題以上に多様な評価指標があります｡そのため､｢その性能指標が高い数値になればなるほど､自分の目的はよく達成される｣と言えるような指標を慎重に選ぶ必要が出てきます｡これができなければ､汎化性能を本来のものよりも高く見積もりすぎてしまう危険があります｡

9.1.1 混同行列（2クラス分類の場合）¶

混同行列とは､モデルの間違いの傾向を知るための行列です｡まずは2クラス分類における､陽性と陰性を分類することを考えます｡縦軸に正解ラベル､横軸に予測ラベルをとることで､陽性を陰性と間違えやすいのか､陰性を陽性と間違えやすいのかなどの傾向がわかります｡分類問題の性能指標は､この混同行列をもとに作られるので非常に重要な概念です｡

コラム : 正答率 99%の分類器は優秀か？¶

機械学習のモデルに得意不得意があるように､性能指標にも得意不得意があり､ケースに応じた選択をする必要があります｡実は､今まで当たり前のように使ってきた正答率にも弱点があり､サンプル数の少ないクラスを軽視してしまっても､ある程度の高い数値が得られてしまいます｡

極端な例を挙げると､医療現場における疾病判定をするときに､100人中1人しか罹らない病気の判定に関して､全員に｢健康です｣と診断する分類器をつくれば､正答率は99%になります｡しかし､病気に罹っている一人を見つけ出せないのでは､分類器をつくった意味がありません｡なのであれば､このモデルをつくるときに､正答率が高いだけのモデルをつくろうとしても本質的には意味がないのです｡

9.1.3 再現率（recall）¶

再現率（別名: 感度､ヒット率､真陽性率）は､｢陽性であるサンプルを､いかに見抜けたか｣を表す指標です｡定義式の構造上､偽陰性が少なくなるほど高くなるようになっています｡偽陰性とは､本当は陽性だけど陰性だと予測してしまったtestサンプルの数なので､｢少しでも陽性の可能性があるものは､念のため陽性にしておこう｣という考え方をすると､再現率は高くなることになります｡

利用されるケースとしては､医療現場における疾病の判定などが挙げられます｡たとえば､癌にかかっているかどうかを予測するようなケースです｡この場合は､偽陽性が出てしまっても精密検査の費用が無駄になるだけですが､偽陰性が出てしまうとその患者の早期発見が遅れてしまい､取り返しがつかないほど癌が進行してしまうことになりかねません｡そのため､偽陰性を少なくすることに焦点を当てる再現率の利用が好ましいのです｡

9.1.4 適合率（precision）¶

適合率（別名:陽性的中率）は､｢モデルによる陽性判定が､どの程度信頼できるか｣を表す指標です｡定義式の構造上､偽陽性が少なくなるほど高くなるようになっています｡偽陽性とは､本当は陰性だけど陽性だと予測してしまったtestサンプルの数なので､｢確実に陽性だと思えるものしか､陽性と判定しない｣という考え方をすると､適合率は高くなることになります｡

利用されるケースとしては､検索エンジンによるWebページの選別などが挙げられます｡ユーザーが検索ワードを指定して､それに関連するWebページが陽性だとします｡このとき､ユーザーは自分が欲しかった情報が載っていない､陰性のページばかりが並んでいるとイライラします｡つまりは､陰性がどれだけ混ざり込んでいないかといことが重視されるのです｡そのため､偽陽性を少なくすることに焦点を当てる適合率の利用が好ましいのです｡

多クラス分類の場合の性能指標は､各クラスについて考えた一対他の2値分類の $TP_j, FP_j, TN_j, FN_j$ を利用します｡算出には､ミクロ平均とマクロ平均という計算手法を用います｡両者の違いは､クラスごとのサンプル数の偏りを調整するかしないかです｡調整しないときはミクロ平均を､調整するときはマクロ平均を利用します｡サンプル数が多いクラスを重視すするのか､全てのクラスを平等に扱うのかによって使い分けることができます｡

9.1.7 ミクロ平均¶

ミクロ平均とは､クラスごとのサンプル数の偏りを調整しない場合の平均の取り方です｡たとえば､$TP$ を利用しているところを $TP_0 + ... + TP_n$ に置き換えて計算します｡たとえば､適合率のミクロ平均は以下のような計算式になります｡

$$PRE_{micro} = \frac{TP_0 + ... + TP_n}{(TP_0 + FP_0)+ ... + (TP_n + FP_n)}$$

9.1.8 マクロ平均¶

マクロ平均とは､サンプル数の偏りを調整する場合の平均の取り方です｡各クラスごとに一対他の2値分類を考え､各クラスごとの $TP_j, FP_j, TN_j, FN_j$ を求めるところまではミクロ平均と同じですが､マクロ平均では各クラスのごとに個別に評価指標を求めてしまい､その後で性能指標の平均を取ります｡たとえば､適合率のマクロ平均は以下のような式になります｡

$$PRE_{macro} = \frac{PRE_0 +...+ PRE_n}{n}$$

なぜ､マクロ平均のような計算の工夫がされるのかというと､ミクロ平均では､サンプル数の少ないクラスが軽視されてしまうためです｡クラスごとのサンプル数におおきな偏りがあったとき､ミクロ平均で考えてしまうと､サンプル数が多いクラスに対してさえ上手く予測できていれば､少ないクラスに対して外してしまってもさほど影響がないという状態になってしまいます｡しかし､マクロ平均のこのように先に評価指標を計算すれば､サンプルごとの個数の偏りが性能指標の計算のときに吸収されて､平均の数値に影響しにくくなります｡

以上のことを踏まえて､ミクロ平均とマクロ平均を使い分ける必要があります｡多数派のクラスを重視したいときはミクロ平均を､全体を均等に評価したいときはマクロ平均を利用するのがよいでしょう｡

9.2 交差検証法¶

交差検証法（Cross_Validation）とは､アルゴリズムが実現するであろう汎化性能を信頼性高く見積もる手法です｡学習データをk個に分割して､うち1つを検証用､残りを学習用にしてホールド・アウト法の学習を行い､それをk回繰り返します｡そして､k回の学習のそれぞれでtest精度を求めた後にそれらの平均を取り､モデルの実現しうるtest精度とします｡このことから､交差検証はk分割交差検証法（k-fold Cross-Validation）とも呼ばれています｡通常のホールド・アウト法よりも信頼性の高い評価ができることが利点です｡

9.2.1 ホールド・アウト法の問題点¶

これまで利用してきたホールド・アウト法には､検証データの選び方によってtest精度が大きくばらついてしまう可能性が高いという欠点があります｡たとえば､分割の際にたまたま予測がしにくいデータばかりがtestデータに割り当てられてしまうと､test精度は本来あるべき値より低く見積もられます｡逆に､余計に高く見積もられることもあります｡特に､規模の小さいデータほどその可能性が高くなります｡しかし､交差検証法を利用するとtestデータを取り替えた場合のそれぞれについて平均をとるので､こうした偶然によるばらつきを吸収することができます｡

KFold¶

データの分割方法の指定を､sklearn.model_selectionのKFoldを利用して実行できます｡各引数の意味合いは以下の通りです｡

  • n_splits: データを分割する個数k｡通常5〜10個程度｡小さいデータほど数多く分割する必要がある｡
  • shuffle: 分割を行う前にデータをシャッフルするかどうか｡Trueにしておいた方が信頼性が上がる｡
  • random_state: 乱数生成器の状態｡これを固定しておけば､毎回同じ分割結果が得られる｡

9.3 グリッドサーチ¶

グリッドサーチとは､ハイパーパラメータのちょうどいい値を探索的に定める手法です｡その名の通り格子点（Grid）の上を探索（Search）するようにしらみ潰しにハイパーパラメータの組み合わせを試して､汎化性能が最も高くなるような組み合わせを探します｡たとえば､前章で扱ったカーネルSVMであれば､ちょうどいい $C$ と $\gamma$ の組み合わせを見つけることを目指します｡また､今回はこのグリッドサーチをscikit-learnのGridSearchCVクラスを用いて行います｡こちらは､グリッドサーチを交差検証法を用いて行うクラスです｡

9.3.1 前処理¶

今回も､前章と同じXORのデータを利用します｡

ホールド・アウト法による分割¶

グリッドサーチの場合は､あらかじめホールド・アウト法による分割が必要になります｡trainデータとtestデータの分割を行い､trainデータだけを用いてグリッドサーチを行います｡なぜかというと､testデータを含めてハイパーパラメータの組み合わせの選定をしてしまうと､testデータに都合よく適合するパラメータが選ばれてしまうからです｡testデータは将来的に得られるデータを想定しているため､モデルにとって完全に初めて見るデータになっていなければ意味がありません｡なので､ハイパーパラメータの調整にも利用してはいけないのです｡また､このようにtestデータの情報がモデルに漏れ出すなどして､運用の際に利用できない情報がモデルに与えられてしまうことをleakageといいます｡

9.3.2 GridSearchCVの学習¶

GridSearchCVは､その名の通り交差検証を用いたグリッドサーチを実行してくれるクラスです｡また､以下のフローを一気通貫で行ってくれるため､非常に便利です｡多くの場合､交差検証という単語は､こちらのGridSearchCVを指して言われます｡

  1. 指定のハイパーパラメータの組み合わせのすべてについて交差検証を行う
  2. すべての組み合わせについて交差検証の平均値を比較する
  3. 最もtest精度が高かったパラメータを用いてインスタンスを生成する
  4. 引き渡されたインスタンスを学習させる
  5. 以後､学習済みのインスタンスと同様に振る舞う

計算量の課題¶

GridSearchCVは計算時間がかかりやすいという課題を抱えています｡たとえば､ $\gamma$ を5通り､ $C$ を5通り指定して､交差検証を10分割で行うとすれば､単純に5×5×10=250回の学習が必要になります｡その場合は学習時間も単純に通常の250倍かかります｡データが大規模になったり､モデルが複雑になったりするほどこのデメリットは深刻になります｡

# ２次元データの決定曲線をプロットする関数
def plot_decision_regions(X, y, classifier, resolution=0.02):
    from matplotlib.colors import ListedColormap
    markers = ("s", "x", "o", "^", "v")
    colors = ("red", "blue", "lightgreen", "gray", "cyan")
    cmap = ListedColormap(colors[:len(np.unique(y))])
    
    x1_min, x1_max = X[:, 0].min()-1 , X[:, 0].max() +1
    x2_min, x2_max = X[:, 1].min()-1 , X[:, 1].max() +1
    
    xx1, xx2 = np.meshgrid(np.arange(x1_min, x1_max, resolution), np.arange(x2_min, x2_max, resolution))
    Z = classifier.predict(np.array([xx1.ravel(), xx2.ravel()]).T)
    Z = Z.reshape(xx1.shape)
    plt.contourf(xx1, xx2, Z, alpha=0.4, cmap=cmap)
    plt.xlim(xx1.min(), xx1.max())
    plt.ylim(xx2.min(), xx2.max())
    for idx, cl in enumerate(np.unique(y)):
        plt.scatter(x=X[y == cl, 0], y=X[y == cl, 1], alpha=0.8, c=cmap(idx),marker=markers[idx], label=cl)
    plt.show()
